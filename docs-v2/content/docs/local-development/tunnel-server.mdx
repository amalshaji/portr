The tunnel server is built using Go and handles WebSocket tunneling to tunnel HTTP/TCP connections.

## Requirements

- **Go** (1.25+)
- **PostgreSQL** (16+) or **SQLite** (3.0+)

## Setup Process

### Configure environment variables

Create a new `.env` file using the `.env.template` file as a reference.

### Start the server

```shell
go run ./cmd/portrd start tunnel
```

You should see output similar to:

```shell
time=2024-03-29T19:16:35.023+05:30 level=INFO msg="starting WebSocket tunnel endpoint" port=:8001
time=2024-03-29T19:16:35.023+05:30 level=INFO msg="starting proxy server" port=:8001
time=2024-03-29T19:16:35.023+05:30 level=INFO msg="Starting 1 cron jobs"
```

This starts:

- **WebSocket tunnel endpoint** on port `:8001` at `/_portr/tunnel/connect`
- **Proxy server** on port `:8001` for wildcard HTTP, TCP, and upgraded WebSocket traffic

When running the combined server with `go run ./cmd/portrd start all`, the admin
server listens on `:8000` by default and the tunnel/proxy listener remains on
`:8001`.

## Server Architecture

The tunnel server consists of several components:

### WebSocket Tunnel

- Handles incoming tunnel connections from clients
- Manages WebSocket tunneling
- Authenticates clients using tokens

### Proxy Server

- Routes incoming HTTP requests to the appropriate tunnels
- Handles subdomain routing
- Manages SSL termination in production

### Cron Jobs

- Cleans up expired tunnel connections
- Performs maintenance tasks

```bash
go build ./...

go test ./...

# Format code
go fmt ./...

# Run linting
go vet ./...
```

### Configuration

For all configuration variables, check out the [tunnel server config file](https://github.com/amalshaji/portr/blob/main/internal/server/config/config.go).

Key configuration options include:

- **Database connection**: PostgreSQL connection string
- **WebSocket tunnel endpoint settings**: Proxy URL and authentication
- **Proxy settings**: HTTP server configuration
- **Security settings**: Token validation and access control

## Testing Tunnel Connections

<Callout type="info">
  Make sure the admin server is running before testing tunnel connections, as it
  handles client authentication.
</Callout>

### Local Testing

1. Start the admin server
2. Start the tunnel server
3. Build and run the client to create a test tunnel

### Connection Flow

1. Client connects to the WebSocket tunnel endpoint on port 8001
2. WebSocket tunnel endpoint validates the client token with the admin server
3. HTTP tunnels register one or more WebSocket workers; TCP tunnels register one remote TCP listener
4. Proxy server routes incoming requests or TCP bytes to the client

## Troubleshooting

### Common Issues

**Port conflicts**: Ensure port 8001 is available.

**Database connection errors**: Verify your PostgreSQL instance is running and accessible.

**WebSocket connection failures**: Check that the WebSocket tunnel endpoint is properly configured and the client has valid credentials.

### Debugging

Enable tunnel debug logging with:

```bash
PORTR_TUNNEL_DEBUG=true
```

View detailed logs to troubleshoot connection issues:

```bash
# Follow server logs
go run ./cmd/portrd start tunnel
```
